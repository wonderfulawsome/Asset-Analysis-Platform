# 작업 규칙

## 응답 길이
- **모든 설명은 한 줄로 한다.** 부연 없이 핵심 한 문장.
- 표/코드 블록을 요청한 경우는 예외 (그래도 본문 설명은 한 줄)

## 커밋·푸시 확인
- **구현(코드 변경) 이 끝나면 반드시 사용자에게 "커밋·푸시할까요?" 물어본다.** 자동으로 커밋/푸시하지 말 것.
- structure.md 갱신까지 모두 마친 뒤에 묻는다 (커밋에 같이 들어가야 함).
- 사용자가 명시적으로 "커밋해", "푸시해", "커밋푸시도 해" 라고 한 경우는 묻지 말고 즉시 실행.

## 커밋·푸시 범위 — Claude 가 손댄 파일만
- **다른 터미널/세션이 동시에 작업 중일 수 있다.** 커밋·푸시는 *이 세션에서 Claude 가 직접 편집한 파일만* 포함시킨다.
- `git add .` / `git add -A` / `git add -u` 같은 일괄 스테이징 금지. 항상 **파일 경로를 명시적으로 나열**해 `git add path/to/file1 path/to/file2 ...` 로 추가한다.
- 커밋 직전 `git status --short` 로 의도한 파일만 staged (`M`/`A`) 인지 확인. modified 또는 untracked 인 파일 중 *내가 안 건드린 것* 은 스테이징·커밋·푸시에서 *제외*.
- 캐시 버스트 (`?v=N` bump 등) 가 다른 세션 변경과 같은 파일 (예: `templates/stocks.html`) 안에 섞여 있으면 — 그 파일을 통째 add 하지 말고, **내 변경 라인만 골라서 적용**하거나 새 cleanup 커밋으로 비-내 변경을 제거한다.
- 사용자가 "전부 커밋해" 같은 명시적 일괄 지시를 내리면 그때만 일괄 add 가능. 기본은 명시적 화이트리스트.

## 구현 완료 보고 형식
- **구현이 끝나면 항상 아래 3 블록을 모두 포함해 *자세히* 설명한다.** 한 줄 짜리 압축 보고는 금지 — 각 블록을 충분한 깊이로 풀어 쓴다.
  1. **[변경 위치]** — 파일 경로 + line 번호(또는 함수명) + **변경 후 위치·로직 설명만** (코드 스니펫·Before/After diff 모두 금지). 어떤 함수/블록을 어떻게 바꿨는지 자연어로 묘사 + 변경 사유 한 줄. 파일 여러 개면 각각 별도 항목으로.
  2. **[변경 결과]** — 사용자가 화면/동작에서 관찰 가능한 변화. **수치 예시 또는 실제 응답 샘플** 1~2건 포함. edge case 가 있다면 어떻게 처리되는지도 명시 (예: "데이터 결손 시 history 최근값으로 폴백").
  3. **[최종 파이프라인]** — 변경 후 전체 흐름. **단계별 번호 리스트**로 작성, 각 단계에 어떤 함수/엔드포인트가 무엇을 하는지 포함. 변경된 단계는 굵게 표시. 입력 → 처리 → 캐시 → 응답 → 렌더 단계를 빠짐없이.
- 추가로 다음 항목들이 해당하면 별도 블록으로:
  - **[edge case / fallback]** — None / 빈 데이터 / 옛 캐시 / 외부 API 실패 시 동작
  - **[검증]** — syntax check 결과, 실제 endpoint probe 결과 (HTTP 코드 + 응답 일부), 로컬 verify 못 한 경우 그 이유
  - **[제한 / 추후 작업]** — 이번 변경으로 해결되지 않은 부분, 의존 데이터 결손 등
- 한 줄 응답 규칙은 일반 대화에 적용. 구현 보고는 **자세함 우선**, 압축하지 말 것.
- 이 형식은 "구현 후 보고" 에만 적용 — 단순 질문 답변·탐색·설명 등에는 적용 안 함.

## update.py 갱신 금지
- **`update.py` 는 더 이상 갱신하지 않는다.** 코드 변경 후 새 `[N]` 엔트리를 추가하지 말 것.
- 기존 엔트리 ([1]~[126]) 도 건드리지 말고 history archive 로만 둔다.

## logic/structure.md 갱신 (단독 운영)
- 코드 변경이 발생하면 `logic/structure.md` 만 갱신한다 (update.py 동반 갱신 X).
- structure.md = 현재 시점 청사진 (시스템이 지금 어떻게 구성·동작) + 변경 이력의 한 줄 요약을 마지막에 누적
- 새 컴포넌트·엔드포인트·테이블·환경변수 추가 → structure.md 의 해당 섹션 반영
- 함수 시그니처·알고리즘 변경 → structure.md 의 해당 섹션 업데이트
- 마지막에 "마지막 갱신 시점" 줄을 최신 작업 요약으로 갱신 (한 줄, 사유·구현 요지·검증 포함)

## 데이터 분석 결정 — 묻기 vs 직접 결정

다음 종류의 결정에 도달하면 **코드를 작성하기 전에 `AskUserQuestion`** 으로 사용자에게 물어본다:

1. 데이터 범위·기간·필터링 기준
2. 이상치·결측치 처리 방식
3. 타겟 변수 정의
4. 피처 선택 시 의미적 판단이 필요한 경우
5. 모델 선택의 트레이드오프 (해석력 vs 정확도 등)
6. 평가 지표·검증 방법
7. 임계값(threshold) 설정
8. 사용자 페르소나·세그먼트 정의

반대로 다음은 묻지 말고 직접 결정한다:
- 코드 구조, 변수명, 라이브러리 선택
- 명백한 베스트 프랙티스
- 단순 버그 수정·리팩토링

**판단 후엔 "왜 이 옵션을 추천하는지" 짧게 근거도 같이 제시한다.**

## 투자자문 리스크 회피 (웹 서비스 산출물 한정)
- **적용 범위**: 이 규칙은 **개발 중인 웹 서비스의 사용자 노출 산출물**(UI 텍스트, 리포트, 알림, API 응답, 분석 결과 화면, 마케팅 카피 등)에만 적용된다. 채팅창 내 설명·기획 논의에는 적용하지 않는다.
- 웹 서비스 산출물은 **수익화를 위한 투자자문으로 비칠 수 있는 표현을 항상 배제**한다 (자본시장법상 투자자문업 미등록 리스크 회피 목적).
- 금지: "사라/팔아라", "지금이 매수 타이밍", 목표주가 제시, 특정 종목 수익률 보장·암시, 포트폴리오 추천
- 허용: 공개 지표 표시, 객관적 데이터 시각화, 일반적 금융 개념 설명, 사용자가 직접 입력한 조건의 백테스트 결과
- 종목·ETF 관련 화면을 만들 땐 **사실(지표·뉴스) → 데이터 해석 → 면책 문구** 구조로 구성하고, 화면 어딘가에 **"본 정보는 투자자문이 아니며 투자 판단과 책임은 사용자에게 있습니다"** 류 면책 문구를 노출한다.
- 코드·문구 작성 시 애매하면 자문성 표현을 빼는 쪽으로 결정한다.
