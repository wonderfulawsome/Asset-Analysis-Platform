# 작업 규칙

## 응답 길이
- **모든 설명은 한 줄로 한다.** 부연 없이 핵심 한 문장.
- 표/코드 블록을 요청한 경우는 예외 (그래도 본문 설명은 한 줄)

## update.py 자동 갱신
- 코드 변경 작업이 끝날 때마다 **반드시** `update.py`에 새 `[N]` 엔트리를 추가한다.
- 사용자가 따로 요청하지 않아도 자동으로 한다.
- 엔트리 형식 (기존 [9]~[20] 패턴 유지):
  - 헤더: `# [N] YYYY-MM-DD (UTC) — 작업 제목` (가운데 정렬용 ═ 가로줄)
  - `[개요]`, `[신규 파일]`, `[수정 파일]`, `[검증 결과]` 같은 라벨로 구조화
  - 핵심 코드 스니펫은 ` # ` 으로 감싸지 말고 **실제 코드 그대로** 붙여 넣기 (기존 규약)
  - 각 변경의 "왜"(이유·근거)도 함께 기록 — 단순 What 이 아닌 Why

## 시점
- 단일 변경 → 그 자리에서 [N] 엔트리 1개
- 여러 단계 작업(Stage 등) → 각 단계 종료 시점마다 별도 엔트리
- 사용자가 "다 됐어?" 또는 "다음 진행" 류 발화 직전에 update.py 정리되어 있어야 함

## 예외
- update.py 자체 수정만 한 경우는 새 엔트리 불필요
- 단순 typo/주석 수정도 엔트리 생략 가능 (판단)

## logic/structure.md 동시 갱신
- 코드 변경 후 update.py 에 [N] 엔트리 작성과 **동시에** `logic/structure.md` 도 해당 부분을 갱신한다
- update.py = 시간순 변경 이력 (어떤 변경이 언제 왜)
- structure.md = 현재 시점 청사진 (시스템이 지금 어떻게 구성·동작)
- 새 컴포넌트·엔드포인트·테이블·환경변수가 추가되면 structure.md 의 해당 섹션(2~17)에 반영
- 함수 시그니처가 바뀌거나 알고리즘이 변경되면 structure.md 의 해당 섹션 업데이트
- 마지막에 "마지막 갱신 시점" 줄을 최신 작업 요약으로 갱신
