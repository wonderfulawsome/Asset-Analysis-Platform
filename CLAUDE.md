# 작업 규칙

## 응답 길이
- **모든 설명은 한 줄로 한다.** 부연 없이 핵심 한 문장.
- 표/코드 블록을 요청한 경우는 예외 (그래도 본문 설명은 한 줄)

## 커밋·푸시 확인
- **구현(코드 변경) 이 끝나면 반드시 사용자에게 "커밋·푸시할까요?" 물어본다.** 자동으로 커밋/푸시하지 말 것.
- update.py / structure.md 갱신까지 모두 마친 뒤에 묻는다 (커밋에 같이 들어가야 함).
- 사용자가 명시적으로 "커밋해", "푸시해", "커밋푸시도 해" 라고 한 경우는 묻지 말고 즉시 실행.

## update.py 자동 갱신
- 코드 변경 작업이 끝날 때마다 **반드시** `update.py`에 새 `[N]` 엔트리를 추가한다.
- 사용자가 따로 요청하지 않아도 자동으로 한다.
- 엔트리 형식 (기존 [9]~[20] 패턴 유지):
  - 헤더: `# [N] YYYY-MM-DD (UTC) — 작업 제목` (가운데 정렬용 ═ 가로줄)
  - `[개요]`, `[신규 파일]`, `[수정 파일]`, `[검증 결과]` 같은 라벨로 구조화
  - 핵심 코드 스니펫은 ` # ` 으로 감싸지 말고 **실제 코드 그대로** 붙여 넣기 (기존 규약)
  - 각 변경의 "왜"(이유·근거)도 함께 기록 — 단순 What 이 아닌 Why

## 시점
- 단일 변경 → 그 자리에서 [N] 엔트리 1개
- 여러 단계 작업(Stage 등) → 각 단계 종료 시점마다 별도 엔트리
- 사용자가 "다 됐어?" 또는 "다음 진행" 류 발화 직전에 update.py 정리되어 있어야 함

## 예외
- update.py 자체 수정만 한 경우는 새 엔트리 불필요
- 단순 typo/주석 수정도 엔트리 생략 가능 (판단)

## logic/structure.md 동시 갱신
- 코드 변경 후 update.py 에 [N] 엔트리 작성과 **동시에** `logic/structure.md` 도 해당 부분을 갱신한다
- update.py = 시간순 변경 이력 (어떤 변경이 언제 왜)
- structure.md = 현재 시점 청사진 (시스템이 지금 어떻게 구성·동작)
- 새 컴포넌트·엔드포인트·테이블·환경변수가 추가되면 structure.md 의 해당 섹션(2~17)에 반영
- 함수 시그니처가 바뀌거나 알고리즘이 변경되면 structure.md 의 해당 섹션 업데이트
- 마지막에 "마지막 갱신 시점" 줄을 최신 작업 요약으로 갱신

## 데이터 분석 결정 — 묻기 vs 직접 결정

다음 종류의 결정에 도달하면 **코드를 작성하기 전에 `AskUserQuestion`** 으로 사용자에게 물어본다:

1. 데이터 범위·기간·필터링 기준
2. 이상치·결측치 처리 방식
3. 타겟 변수 정의
4. 피처 선택 시 의미적 판단이 필요한 경우
5. 모델 선택의 트레이드오프 (해석력 vs 정확도 등)
6. 평가 지표·검증 방법
7. 임계값(threshold) 설정
8. 사용자 페르소나·세그먼트 정의

반대로 다음은 묻지 말고 직접 결정한다:
- 코드 구조, 변수명, 라이브러리 선택
- 명백한 베스트 프랙티스
- 단순 버그 수정·리팩토링

**판단 후엔 "왜 이 옵션을 추천하는지" 짧게 근거도 같이 제시한다.**

## 투자자문 리스크 회피 (웹 서비스 산출물 한정)
- **적용 범위**: 이 규칙은 **개발 중인 웹 서비스의 사용자 노출 산출물**(UI 텍스트, 리포트, 알림, API 응답, 분석 결과 화면, 마케팅 카피 등)에만 적용된다. 채팅창 내 설명·기획 논의에는 적용하지 않는다.
- 웹 서비스 산출물은 **수익화를 위한 투자자문으로 비칠 수 있는 표현을 항상 배제**한다 (자본시장법상 투자자문업 미등록 리스크 회피 목적).
- 금지: "사라/팔아라", "지금이 매수 타이밍", 목표주가 제시, 특정 종목 수익률 보장·암시, 포트폴리오 추천
- 허용: 공개 지표 표시, 객관적 데이터 시각화, 일반적 금융 개념 설명, 사용자가 직접 입력한 조건의 백테스트 결과
- 종목·ETF 관련 화면을 만들 땐 **사실(지표·뉴스) → 데이터 해석 → 면책 문구** 구조로 구성하고, 화면 어딘가에 **"본 정보는 투자자문이 아니며 투자 판단과 책임은 사용자에게 있습니다"** 류 면책 문구를 노출한다.
- 코드·문구 작성 시 애매하면 자문성 표현을 빼는 쪽으로 결정한다.
